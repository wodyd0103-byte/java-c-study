차트는 Git에, 비밀은 Vault에 — 게시판을 어떻게 보관할까

게시판은 완성됐다. 그런데 이 전부를 어떻게 보관하고, 통째로 재현할까?

03 / 10
선언은 Git · 값은 Vault

Git 저장소 (진실원천 후보)

"무엇을 어떻게 배포하는가"를 글로 적어 둔다 — 사람이 읽고, 버전이 남고, 통째로 재현된다.

board/ Helm 차트 (3티어 템플릿)
values-dev.yaml · values-prod.yaml
externalsecret.yaml (비밀의 위치만)
password: ...  ← 한 줄도 없음
git grep -i password 로 스스로 검증

Vault (비밀 금고)

도전①에서 먼저 지은 금고. 실제 비밀 값은 여기에만 있고, Git엔 들어가지 않는다.

secret/board → mysql 비밀번호 등
ExternalSecret 이 동기화 → K8s Secret
PVC 의 게시판 데이터는 그대로 보존
차트만 install 해도 비밀·데이터는 살아 있음

이것이 GitOps의 비밀 패턴  —  차트·values·ExternalSecret 선언은 Git, 비밀 값은 Vault. 이 저장소가 다음주 ArgoCD가 동기화할 source가 된다

04 / 10

Git에 두는 것 — 재현 가능한 '구조'

Helm 차트 · 환경별 values.yaml · ExternalSecret 선언.
비밀의 위치(이름·경로)만 적고 값은 한 줄도 없음.

Vault에 두는 것 — 흐르는 '값'

DB 암호 등 비밀 값은 secret/board에.
ESO가 클러스터에서 끌어와 Secret을 동기화.

한 명령으로 재현  helm install board ./board-chart
구조는 Git이, 값은 Vault가 — 이 저장소가 다음주 ArgoCD의 source.

board-chart/ git repo
templates/ (nginx·tomcat·mysql) 3티어
values-prod.yaml 환경설정
externalsecret.yaml 위치만
password = ******** 없음
ExternalSecret 선언  ──▶  ESO가 끌어옴
Vault  secret/board
db_password = (평문 값은 여기만)

구조=Git · 값=Vault → 사람이 손으로 동기화하면 진실이 흔들린다

Day 9 · 7교시 · 실습 1

차트는 Git에, 비밀 값은 한 줄도 없이

05 / 10
직접 따라 입력
# 1) 게시판 차트 폴더를 저장소로 초기화 $ cd board-chart/ $ git init && git add . # 2) 차트·values·ExternalSecret 선언을 기록 $ git commit -m "board 3-tier chart" # 3) 비밀 값이 새어들지 않았는지 직접 검증 $ git grep -i 'password'
Git엔 위치만, 값은 Vault에. ExternalSecret은 "어디서 가져올지"만 적힌 선언서 — 평문은 들어있지 않습니다.

// 예상 출력

$ git commit -m "board 3-tier chart"
[main (root-commit) a1c3e9f] board 3-tier chart
9 files changed, 214 insertions(+)

$ git grep -i 'password'
externalsecret.yaml: remoteRef:
  key: board, property: mysql-password
# 키 '이름'만 잡힘 — 실제 값은 없음
확인. grep에 걸린 건 Vault를 가리키는 키 이름뿐. 이 저장소가 곧 ArgoCD가 동기화할 진실의 원천이 됩니다.
Day 9 · 7교시 · 실습 2

통째 재현 — Git의 차트만으로 helm install 부활

06 / 10
직접 입력
# 게시판을 통째로 내린다 (앱은 사라짐) $ helm uninstall board -n board # Git 클론본의 차트 그것만으로 재설치 $ helm install board ./board-chart \ -n board -f values-prod.yaml
새 요소: 실습1은 '올리기'였다. 여기선 내려놓고 Git만으로 되살린다 — 비밀은 Vault, 데이터는 PVC가 그대로 들고 있다.

// uninstall 후 → install, 무엇이 살아남나

$ kubectl get pvc -n board NAME STATUS VOLUME CAPACITY data-board-mysql-0 Bound pvc-9f3c.. 5Gi <- 남아있음
$ helm install board ./board-chart -n board ... NAME: board STATUS: deployed REVISION: 1 $ kubectl get externalsecret -n board NAME STORE STATUS board-secret vault-backend SecretSynced
$ kubectl get pod -n board NAME READY STATUS board-nginx-.. 1/1 Running board-tomcat-.. 1/1 Running board-mysql-0 1/1 Running <- 이전 글 그대로
확인: 차트(Git)만 install 했는데 게시판이 부활했다 — 비밀(Vault)·데이터(PVC)는 차트 밖에서 보존돼 다시 연결됐다.
Day 9 · 7교시 · 🔥 반전 포인트

재현은 됐는데 — 지금 떠 있는 게 Git의 어느 버전입니까?

차트로 게시판을 되살렸다. 그런데 둘이 각자 손으로 고치자, 아무도 진실을 모른다.

07 / 10
손으로 고친 뒤, Git과 비교
$ kubectl edit deploy board-tomcat # replicas 1→3
deployment.apps/board-tomcat edited

$ helm diff upgrade board ./board-chart
board, board-tomcat, Deployment (apps) has changed:
-   replicas: 3  # 클러스터(지금)
+   replicas: 1  # Git(차트)

어긋남(drift)이 생겼다

Git엔 replicas: 1인데 클러스터는 3. 옆 조원도 손으로 고치면 — '지금 뜬 게 어느 버전인지' 추적이 끊긴다.

// 정석: 차트는 Git, 비밀은 Vault

Git: 차트·values·ExternalSecret 선언 (비밀의 위치만)
Vault: 비밀 값 자체. Git엔 한 줄도 없음

재현은 되는데 — 누가 진실인가?

파일이 진실원천이려면, 사람이 손으로 동기화하는 한 그 약속은 깨진다. 다음주 ArgoCD가 Git을 자동으로 클러스터에 맞춘다 — 이 저장소가 그 source다.

Day 9 · 7교시 · 실습 3

어긋남(drift) 만들기 — 지금 뜬 게 Git의 어느 버전인가

08 / 10
손으로 클러스터를 고친다
# Git엔 replicas: 2 / image:1.0 로 커밋돼 있다 $ git log --oneline -1 a1c3e9f board chart v1.0 (replicas 2) # 그런데 운영 중 손으로 만진다 — 두 사람이 각자 $ kubectl edit deploy board-tomcat # replicas→4 $ helm upgrade board ./board --set nginx.image=1.2
차트는 안 건드렸다. Git 파일은 그대로인데, 클러스터만 v1.0에서 멀어진다 — 이 차이가 drift.

// 지금 뜬 게 Git의 어느 버전인지 추적

$ kubectl get deploy board-tomcat -o jsonpath='{.spec.replicas}' 4 # Git은 2 → 누가 언제 4로? 기록 없음 $ helm get values board nginx: image: 1.2 # 차트 파일엔 1.0 — 어긋남 $ git status nothing to commit, working tree clean
재현은 되는데, 누가 진실인가. 파일은 clean인데 클러스터는 v1.0이 아니다. 손으로 동기화하는 한, '지금 떠 있는 게 Git의 어느 버전인지' 아무도 답하지 못한다 → 다음주 ArgoCD가 이 격차를 자동으로 메운다.

[도전②·4/4] 매듭 — 차트는 Git에, 비밀은 Vault에

09 / 10
CHALLENGE · 진실원천 봉인 작전

문제. 완성된 게시판 전체(차트·환경별 values·ExternalSecret 선언)를 Git 저장소 하나로 봉인하되, '비밀 값'은 단 한 줄도 Git에 새지 않음을 스스로 증명하라. 그 저장소만으로 빈 namespace에서 게시판이 되살아나는지 확인하고, 사람이 손으로 동기화할 때 무엇이 깨지는지를 근거로 자동화 도구가 왜 필요한지 논증하라.

  • 1봉인. 차트·values·ExternalSecret 선언을 git에 올리고, 비밀 값 0줄을 스스로 검증한다.
  • 2부활. 빈 namespace(또는 옆 조)에 그 저장소만으로 게시판을 재현한다.
  • 3균열. 손으로 클러스터를 고쳐 drift를 만들고, '지금 뜬 게 Git의 어느 버전인지'를 추적한다.

// 출력 예시 (여기까지만 제공)

$ git grep -i password (아무 줄도 나오지 않는다 — Git엔 위치만) $ kubectl get externalsecret -n revive NAME STORE STATUS board-secret vault-backend SecretSynced $ helm install board ./board -n revive … nginx EXTERNAL-IP 로 게시판이 되살아난다
힌트 한 줄. 비밀은 Vault에, Git엔 가리키는 선언만. 두 사람이 각자 kubectl edit로 고치면 — 지금 떠 있는 게 Git의 어느 버전인가? 손으로 맞추는 한 진실원천은 흔들린다.

제출: 재현 성공 증거(스크린샷·로그) + 'ArgoCD가 필요한 이유' 한 문단

Day 9 · 7교시 · 정리

차트는 Git에, 비밀은 Vault에 — 그런데 누가 진실인가

10 / 10

차트·values·ExternalSecret = Git

게시판 3티어 전부가 텍스트 선언. git grep -i password 해도 비밀 값 0줄 — Git엔 '위치'만 적힌다.

비밀 값 = Vault

실제 값은 금고 안. ExternalSecret이 동기화해 Secret 생성. 차트만으로 helm install → 데이터는 PVC·Vault가 보존.

손으로 고치면 drift

둘이 각자 kubectl edit 하면 지금 뜬 게 Git 어느 버전인지 아무도 모른다 — 진실원천이 흔들린다.

drift 직면 — Git과 클러스터가 어긋났다
# 사람이 손으로 replica를 바꿈 (Git엔 1)
$ kubectl edit deployment board-tomcat   # replicas 1 → 3
$ git -C ./board-chart status            nothing to commit, working tree clean
$ kubectl get deploy board-tomcat        READY 3/3  # Git은 1, 클러스터는 3 ⚠️ 누가 맞나?
